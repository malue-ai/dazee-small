# 缓存架构决策：两层 vs 三层

> **决策时间**: 2026-01-19  
> **决策结果**: **两层架构（内存 → PostgreSQL）**  
> **理由**: 简单、高效、符合实际场景

---

## 一、架构对比

### 三层架构（内存 → MemoryDB/Redis → PostgreSQL）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        三层缓存架构                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  用户请求 → APIService → SessionManager                                 │
│                       ↓                                                 │
│  1. 检查内存缓存（纳秒级）                                               │
│     ├─ 命中 → 直接返回                                                  │
│     └─ 未命中                                                           │
│                       ↓                                                 │
│  2. 查询 MemoryDB (Redis ZRANGE)                                        │
│     ├─ 命中 → 加载到内存 → 返回                                          │
│     └─ 未命中                                                           │
│                       ↓                                                 │
│  3. 查询 PostgreSQL（冷启动）                                           │
│     ├─ 加载到内存                                                       │
│     └─ 更新 MemoryDB 缓存 (ZADD)                                        │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

**优点**：
- ✅ 多服务器共享缓存（MemoryDB 是共享的）
- ✅ 减少数据库查询压力
- ✅ 冷启动性能更好（Redis 比 PostgreSQL 快）

**缺点**：
- ❌ 增加架构复杂度（需要维护 Redis 缓存）
- ❌ 缓存一致性管理复杂（写入时需要同步更新三层）
- ❌ 增加运维成本（Redis 缓存失效、内存管理）
- ❌ 可能过度设计（如果会话粘性保证，多服务器共享缓存价值有限）

---

### 两层架构（内存 → PostgreSQL）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        两层缓存架构（推荐）                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  用户请求 → APIService → SessionManager                                 │
│                       ↓                                                 │
│  1. 检查内存缓存（纳秒级）                                               │
│     ├─ 命中 → 直接返回（99% 场景）                                        │
│     └─ 未命中（冷启动，<1% 场景）                                        │
│                       ↓                                                 │
│  2. 查询 PostgreSQL（直接回源）                                           │
│     ├─ 加载到内存                                                       │
│     └─ 返回结果                                                         │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

**优点**：
- ✅ **架构简单**：只有两层，易于理解和维护
- ✅ **维护成本低**：不需要管理 Redis 缓存一致性
- ✅ **性能足够**：内存缓存覆盖 99% 场景（纳秒级）
- ✅ **符合实际**：会话粘性保证，冷启动场景少
- ✅ **数据一致性**：直接查数据库，无需担心缓存不一致

**缺点**：
- ⚠️ 冷启动时直接查数据库（但场景少，影响有限）
- ⚠️ 多服务器无法共享缓存（但会话粘性已解决）

---

## 二、场景分析

### 2.1 会话粘性场景（主要场景，99%+）

**假设**：负载均衡层保证会话粘性（一致性哈希）

```
用户 A 的会话 → 始终路由到 Server 1
用户 B 的会话 → 始终路由到 Server 2
```

**两层架构表现**：
- ✅ 首次请求：冷启动，查询 PostgreSQL（1 次）
- ✅ 后续请求：内存缓存命中（纳秒级，无需查询数据库）
- ✅ **性能表现**：99%+ 请求从内存读取，性能极佳

**三层架构表现**：
- ✅ 首次请求：查询 Redis → 未命中 → 查询 PostgreSQL → 更新 Redis
- ✅ 后续请求：内存缓存命中（与两层架构相同）
- ⚠️ **额外开销**：需要维护 Redis 缓存，但收益有限

**结论**：在会话粘性场景下，**两层架构已经足够**，三层架构的额外收益有限。

---

### 2.2 冷启动场景（<1%）

**触发条件**：
- 服务器重启
- 扩缩容（新服务器）
- 会话粘性失效

**两层架构**：
- 直接查询 PostgreSQL（1 次数据库查询）
- 加载到内存（后续请求从内存读取）

**三层架构**：
- 查询 Redis → 未命中 → 查询 PostgreSQL → 更新 Redis
- 加载到内存（后续请求从内存读取）

**性能对比**：
- 两层：1 次 PostgreSQL 查询（~10-50ms）
- 三层：1 次 Redis 查询（~1-2ms）+ 1 次 PostgreSQL 查询（~10-50ms）+ 1 次 Redis 写入（~1-2ms）

**结论**：冷启动场景少（<1%），三层架构的性能提升有限，但增加了复杂度。

---

### 2.3 多服务器场景

**问题**：如果会话粘性失效，用户请求被路由到新服务器，新服务器没有内存缓存。

**两层架构**：
- 新服务器直接查询 PostgreSQL（冷启动）
- 加载到内存（后续请求从内存读取）

**三层架构**：
- 新服务器查询 Redis（可能命中，如果其他服务器已缓存）
- 如果命中：从 Redis 加载（避免查询数据库）
- 如果未命中：查询 PostgreSQL → 更新 Redis

**收益分析**：
- ✅ 三层架构：多服务器可以共享 Redis 缓存
- ⚠️ 但前提是：会话粘性失效频率高（实际场景中很少发生）
- ⚠️ 即使失效，直接查数据库也是可接受的（冷启动是正常场景）

**结论**：多服务器共享缓存的收益有限，不值得增加复杂度。

---

## 三、决策建议：两层架构（推荐）✅

### 3.1 推荐理由

1. **简单性原则**
   - 两层架构更简单，易于理解和维护
   - 减少缓存一致性管理的复杂度
   - 降低运维成本

2. **性能足够**
   - 内存缓存覆盖 99%+ 场景（纳秒级访问）
   - 冷启动场景少（<1%），直接查数据库可接受
   - PostgreSQL 查询性能已优化（复合索引）

3. **符合实际场景**
   - 会话粘性保证（负载均衡一致性哈希）
   - 冷启动是正常场景，无需过度优化
   - 数据一致性更重要（直接查数据库，无需担心缓存不一致）

4. **成本效益**
   - 两层架构：简单、维护成本低
   - 三层架构：复杂、维护成本高，但收益有限

---

## 四、架构决策总结

### 最终决策：两层架构 ✅

| 维度 | 两层架构 | 三层架构 | 推荐 |
|------|---------|---------|------|
| **架构复杂度** | 简单 | 复杂 | ✅ 两层 |
| **维护成本** | 低 | 高 | ✅ 两层 |
| **性能（99% 场景）** | 纳秒级（内存） | 纳秒级（内存） | ✅ 相同 |
| **性能（1% 冷启动）** | 10-50ms（PostgreSQL） | 12-54ms（Redis + PostgreSQL） | ✅ 两层（更简单） |
| **数据一致性** | 高（直接查数据库） | 中（需要管理缓存一致性） | ✅ 两层 |
| **多服务器共享** | 不支持（但会话粘性已解决） | 支持（但收益有限） | ⚠️ 平手 |

**结论**：✅ **两层架构更符合实际场景，推荐采用**
