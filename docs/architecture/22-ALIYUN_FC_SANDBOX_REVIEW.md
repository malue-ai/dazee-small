# FC Sandbox 预览与发布功能 - 产品评审

> 版本：1.0  
> 日期：2026-01-20

---

## 一、背景与问题

当前沙盒环境存在以下限制：

| 问题 | 影响 |
|------|------|
| **生命周期短** | 沙盒仅支持30天，超时后无法使用 |
| **无法持久化发布** | 用户生成的应用最大支持24小时,无法长期保存 |
| **文件保存问题** | 环境丢失后，系统文件也会丢失 |

---

## 二、目标

### 第一阶段：实时预览

| 目标 | 描述 | 验收标准 |
|------|------|----------|
| **接入 FC Sandbox** | 完成阿里云 FC Sandbox 的基础接入 | 能够分配独立沙盒环境 |
| **实时预览** | Agent 生成应用代码后，通过 SSE 事件通知前端，在侧边栏 iframe 实时预览 | 代码生成后 3s 内可预览 |
| **支持迭代修改** | 用户可通过对话输入修改应用，预览实时更新 | 用户说"改成红色"，预览自动刷新 |

### 第二阶段：持久化发布

| 目标 | 描述 | 验收标准 |
|------|------|----------|
| **持久化发布** | 将用户代码打包到容器中，生成长期稳定的公开链接 | 生成可分享的永久 URL |

### 第三阶段：数据连接与版本管理

| 目标 | 描述 |
|------|------|
| **数据连接** | Vibe Coding 生成的应用支持数据源和本体论连接 |
| **多次发布** | 支持应用版本管理，可多次迭代发布 |

## 四、方案概述

### 4.1 核心挑战与解法

| 挑战 | 问题描述 | FC Sandbox 解法 |
|------|----------|-----------------|
| **冷启动延迟** | 函数冷启动对实时对话不可接受 | 函数池预热，提前准备好计算资源 |
| **状态保持与隔离** | 每个聊天是独立会话，需严格隔离 | 会话亲和，同一用户请求路由到同一实例 |
| **海量会话扩展** | 需支持并发聊天会话 | 函数池 + 弹性扩缩容 |
| **动态数据存储** | 每个会话产生的文件需持久化 | OSS 动态挂载，数据持久化存储 |

### 4.2 两阶段架构

```
┌─────────────────────────────────────────────────────────────────────────┐
│  阶段一：预热与池化（后台定时任务）                                        │
│                                                                         │
│   createFunction → getFunction(等待Active) → getTrigger → createSession │
│                                    ↓                                    │
│                              存入数据库（status=空闲）                    │
└─────────────────────────────────────────────────────────────────────────┘
                                     ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  阶段二：分配与服务（用户请求触发）                                        │
│                                                                         │
│   用户新建聊天 → 查询空闲函数 → createSession(绑定+OSS挂载) → 返回预览URL  │
│                                    ↓                                    │
│                              更新数据库（status=已分配）                  │
└─────────────────────────────────────────────────────────────────────────┘
```

### 4.3 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| **Session TTL** | 最大 6 小时（21600 秒） | FC 限制，超时后需重新创建 Session |
| **会话亲和 Header** | `x-fc-session-id` | 请求时需携带此 Header 路由到同一实例 |
| **函数规格** | 建议 1C4G 以上 | Sandbox 场景推荐配置 |
| **实例空闲时间** | 建议 1 小时 | 无请求后实例存活时间 |

### 4.4 数据持久化方案（OSS 动态挂载）

| 项目 | 说明 |
|------|------|
| **存储位置** | 阿里云 OSS，按用户/会话隔离目录 |
| **挂载方式** | createSession 时动态挂载 OSS 到容器内 `/mnt/workspace` |
| **使用方式** | Agent 像操作本地文件一样读写 `/mnt/workspace`，数据自动同步到 OSS |
| **数据恢复** | Session 过期后，新实例自动挂载相同 OSS 路径，数据不丢失 |
| **路径规则** | `bucket://{user_id}/{conversation_id}/` |

### 4.5 持久化挂载配置（重要）

> **问题**：动态挂载与 Session 绑定，Session 过期后挂载信息会丢失，新实例无法访问用户数据。

> **解法**：在实例 **PreStop Hook** 中调用 `updateFunction`，将 OSS 挂载配置写入函数级别，实现永久挂载。

| 时机 | 操作 |
|------|------|
| createSession 后 | ❌ 不要立即调用 updateFunction（会导致实例轮转） |
| PreStop Hook 中 | ✅ 调用 updateFunction 持久化 OSS 配置 |

## 六、核心流程

### 6.0 用户完整交互流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            用户完整交互流程                                   │
└─────────────────────────────────────────────────────────────────────────────┘

    ┌──────────┐
    │  用户输入  │  "帮我做一个计算器应用"
    └────┬─────┘
         │
         ▼
    ┌──────────┐
    │  Agent   │  理解需求 → 生成代码 → 写入 Sandbox
    │  处理    │
    └────┬─────┘
         │
         ▼
    ┌──────────────────────┐
    │  前端展示预览页面      │  侧边栏 iframe 加载预览 URL
    │  (SSE: preview_ready) │
    └────┬─────────────────┘
         │
         ▼
    ┌──────────┐     ┌─────────────────────────────────────┐
    │  用户查看  │ ──▶│           用户满意吗？              │
    │  预览效果  │     └──────────┬──────────────┬──────────┘
    └──────────┘                 │              │
                                 │ 不满意        │ 满意
                                 ▼              ▼
                    ┌─────────────────┐   ┌─────────────────┐
                    │  用户输入修改    │   │  点击「发布」    │
                    │ "把按钮改成蓝色" │   │   按钮          │
                    └────────┬────────┘   └────────┬────────┘
                             │                     │
                             ▼                     ▼
                    ┌─────────────────┐   ┌─────────────────┐
                    │  Agent 修改代码  │   │  打包 + 部署    │
                    │  更新 Sandbox   │   │  生成永久 URL   │
                    └────────┬────────┘   └────────┬────────┘
                             │                     │
                             ▼                     ▼
                    ┌─────────────────┐   ┌─────────────────┐
                    │  预览自动刷新    │   │  用户获得分享链接 │
                    │  (多轮修改循环)  │   │  (流程结束)      │
                    └────────┬────────┘   └─────────────────┘
                             │
                             │ 继续循环
                             ▼
                        返回「用户查看预览效果」

```

**流程说明**：

| 阶段 | 用户操作 | 系统响应 |
|------|----------|----------|
| **1. 初始输入** | 用户描述需求："帮我做一个计算器" | Agent 生成完整应用代码 |
| **2. 预览展示** | - | 前端侧边栏显示实时预览 |
| **3. 评估效果** | 用户查看预览，决定是否满意 | - |
| **4a. 不满意 → 修改** | 输入修改需求："按钮太小了" | Agent 修改代码，预览自动刷新 |
| **4b. 满意 → 发布** | 点击「发布」按钮 | 系统打包部署，返回永久链接 |
| **5. 多轮迭代** | 重复 3→4a，直到满意 | 每次修改后预览实时更新 |

> **产品定义**：发布功能的具体交互（如确认弹窗、进度展示、链接复制等）由产品设计确定。

### 6.1 预览流程

```
用户                前端                后端               算法(Agent)          FC Sandbox
 │                   │                   │                   │                   │
 │── 输入需求 ──────▶│                   │                   │                   │
 │                   │── 发送请求 ──────▶│── 转发请求 ──────▶│                   │
 │                   │                   │                   │                   │
 │                   │                   │                   │── 生成代码 ──────▶│
 │                   │                   │                   │                   │ 写入文件
 │                   │                   │                   │                   │
 │                   │◀── SSE: preview_ready + URL ─────────│                   │
 │                   │                   │                   │                   │
 │                   │── iframe 加载预览 URL ───────────────────────────────────▶│
 │                   │                   │                   │                   │
 │◀── 看到预览 ──────│◀──────────────────────────────────────────────────────────│
 │                   │                   │                   │                   │
```

**关键步骤说明**：

| 步骤 | 说明 |
|------|------|
| 1. 用户输入需求 | 用户在对话框描述想要的应用，如"帮我做一个计算器" |
| 2. Agent 生成代码 | Agent 理解需求，生成 HTML/JS/CSS 代码 |
| 3. 写入 Sandbox | 代码写入 Sandbox 文件系统（已挂载 OSS，自动持久化） |
| 4. SSE 通知前端 | 发送 `preview_ready` 事件，携带预览 URL |
| 5. 前端加载预览 | 侧边栏 iframe 加载预览 URL，用户看到效果 |

### 6.2 迭代修改流程

```
用户："把按钮改成蓝色"
    ↓
Agent 理解修改意图
    ↓
修改 Sandbox 中的代码文件
    ↓
SSE 通知前端刷新
    ↓
侧边栏自动刷新，显示更新后的效果
```

### 6.3 会话恢复流程

```
用户关闭页面后再次打开同一对话
    ↓
系统检测到该对话已有 OSS 数据
    ↓
分配新的 Sandbox 实例（可能有冷启动）
    ↓
自动挂载原有 OSS 路径
    ↓
用户继续在原有代码基础上开发（数据不丢失）
```

> **注意**：会话恢复场景会有冷启动（几百 ms ~ 几秒），建议前端显示"正在恢复环境..."提示，消除用户等待焦虑。

### 6.4 会话退出与沙盒回收流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          会话活跃性检测机制                                   │
└─────────────────────────────────────────────────────────────────────────────┘

    ┌──────────────────────────────────────────────────────────────────┐
    │                     会话活跃状态判断                              │
    │                                                                  │
    │   活跃条件（满足任一）：                                          │
    │   • 前端 WebSocket/SSE 连接存活                                  │
    │   • 最近 N 分钟内有用户操作（发消息、修改代码）                     │
    │   • 前端定期发送心跳（heartbeat）                                 │
    │                                                                  │
    │   不活跃判定：                                                    │
    │   • 连接断开 + 超过空闲超时时间（如 30 分钟）                      │
    └──────────────────────────────────────────────────────────────────┘
```

**场景一：用户主动退出**

```
用户关闭浏览器/切换页面
    ↓
前端发送 beforeunload 事件 或 WebSocket 断开
    ↓
后端标记会话状态为「不活跃」
    ↓
启动空闲计时器（如 30 分钟）
    ↓
┌─────────────────────────────────────┐
│  计时器到期前用户是否返回？          │
└──────────┬──────────────┬───────────┘
           │ 是           │ 否
           ▼              ▼
    ┌─────────────┐  ┌─────────────────┐
    │ 取消回收    │  │ 执行沙盒回收     │
    │ 继续服务    │  │ 释放计算资源     │
    └─────────────┘  │ OSS 数据保留     │
                     └─────────────────┘
```

**场景二：超时自动回收**

```
后台定时任务扫描（如每 5 分钟）
    ↓
检查所有会话的最后活跃时间
    ↓
筛选出超过空闲阈值的会话
    ↓
┌───────────────────────────────────────┐
│  对每个超时会话：                      │
│  1. 调用 FC deleteSession 释放实例    │
│  2. 更新数据库状态为「已回收」         │
│  3. 函数实例归还函数池（可复用）       │
│  4. OSS 数据保留（支持后续恢复）       │
└───────────────────────────────────────┘
```

**关键参数设计**：

| 参数 | 建议值 | 说明 |
|------|--------|------|
| **心跳间隔** | 30 秒 | 前端定期发送心跳，维持活跃状态 |
| **空闲超时** | 30 分钟 | 无操作后触发沙盒回收 |
| **扫描周期** | 5 分钟 | 后台定时检查超时会话 |
| **OSS 保留期** | 30 天 | 与沙盒生命周期一致，支持会话恢复 |

**数据库状态流转**：

```
┌─────────┐     分配      ┌─────────┐    用户退出    ┌─────────┐
│  空闲   │ ───────────▶ │  已分配  │ ────────────▶ │  不活跃  │
│ (idle)  │              │ (active) │               │(inactive)│
└─────────┘              └─────────┘               └────┬────┘
     ▲                        ▲                         │
     │                        │ 用户返回                 │ 超时
     │                        └─────────────────────────┤
     │                                                  ▼
     │                  回收函数实例              ┌─────────┐
     └───────────────────────────────────────── │  已回收  │
                                                │(recycled)│
                                                └─────────┘
```

> **重要**：沙盒回收只释放计算资源（FC 实例），OSS 数据保留，用户再次进入时可通过「会话恢复流程」继续开发。

## 七、用户交互场景

| 场景 | 用户操作 | 系统响应 | 
|------|----------|----------|
| 开始对话 | 用户新建聊天 | 后台自动分配 Sandbox + OSS 目录 |
| 生成应用 | 用户描述需求 | Agent 生成代码，侧边栏显示预览 |
