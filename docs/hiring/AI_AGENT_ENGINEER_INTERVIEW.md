## 题目一：Context Engineering（上下文工程）

### 背景

在构建长时运行（Long-running）的 AI Agent 时，Context Window 管理是核心挑战。Anthropic 提出了 **"More context ≠ more intelligence"** 的观点，过大的 Context 不仅增加成本，还会导致注意力稀释和任务完成率下降。

### 场景描述

一个 Agent 需要执行以下任务：
1. 调用搜索工具获取 **10 篇文章**（每篇约 2000 字符）
2. 调用数据分析工具分析一个 **100MB 的 CSV 文件**（返回统计摘要约 50KB）
3. 调用浏览器工具访问 **5 个网页**（每个返回约 100KB HTML）
4. 基于上述信息生成一份报告

### 问题

**1. 问题诊断（10 分）**

分析上述场景中，如果直接将所有工具返回结果放入 Context，会遇到什么问题？请从以下维度量化分析：
- Token 消耗与成本
- LLM 注意力和任务完成率
- 响应延迟
- Cache 利用率

**2. 方案设计（25 分）**

设计一套完整的 Context Engineering 方案，需要包含以下三个策略：

- **Context Reduction（精简）**：如何处理不同类型的工具返回？核心设计原则是什么？
- **Context Offloading（卸载）**：如何设计分层工具架构？各层适合什么场景？
- **Cache Optimization（缓存优化）**：如何设计 System Prompt 结构以提高 Cache 命中率？

**3. 权衡分析（15 分）**

你的精简方案可能导致 LLM 需要更多轮次才能获取完整信息。请分析：
- "减少 Context 占用"与"减少 LLM 调用次数"之间的矛盾如何平衡？
- 在什么场景下应该保留更多信息，什么场景下应该激进精简？
- 如何量化评估精简策略的 ROI？

## 题目二：RAG/Memory 机制优化


### 背景

现代 AI Agent 需要"记忆能力"来实现个性化和持续学习。与传统 RAG 不同，Agent Memory 需要处理多层次、多类型的记忆，并在对话中动态应用。

### 场景描述

一个企业级 AI 助手需要：
1. 记住用户的偏好（如"喜欢简洁的报告"）
2. 在长对话中保持一致性（不忘记之前讨论的内容）
3. 从历史对话中学习用户习惯
4. 处理用户显式上传的"记忆卡片"
5. 检测并解决记忆冲突（如用户偏好变化）

### 问题

**1. 架构设计（20 分）**

设计一个多层记忆架构，需要说明：
- 记忆如何分层？（短期/长期/语义记忆的定义和边界）
- 每层记忆的存储方式、生命周期、访问模式是什么？
- 记忆之间如何关联？检索时如何融合不同层次的记忆？

**2. 核心问题解决（15 分）**

针对以下两个挑战，提供解决方案：

**挑战 A："幻觉式遗忘"**
- LLM 可能"相信"自己记得某些信息，但实际上 Context 中没有
- 如何设计协议强制 LLM 从 Memory 读取，而非依赖 Thinking？
- System Prompt 中应该如何约束 LLM 的行为？

**挑战 B：记忆冲突**
- 用户说"我喜欢 Python"，后来又说"我现在更喜欢 Rust"
- 如何检测这类冲突？检测的信号是什么？
- 冲突发生时，应该采用什么解决策略？

**3. Prompt 注入设计（15 分）**

设计一个将用户画像（Persona）注入 System Prompt 的方案：
- 如何选择注入哪些记忆？（避免 Context 膨胀的选择标准）
- 记忆内容应该如何格式化？（让 LLM 能有效利用的结构）
- 敏感信息（如密码、API Key）如何处理？

## 题目三：Agent Tools 实现机制


### 背景

Claude 推出了 **Skills** 功能，代表了一种新的 Agent 工具设计范式：
- **Traditional Tools**：单一函数调用，最小化描述，每次都全量加载
- **Claude Skills**：完整工作流，包含指令文档（SKILL.md）、可执行脚本、资源文件

Skills 采用 **Progressive Disclosure（渐进式披露）** 架构：Stage 1 只加载简短描述，Stage 2 按需加载完整内容。

### 问题

**1. Progressive Disclosure 设计（15 分）**

- 请描述 Progressive Disclosure 的完整实现流程（从用户请求到 Skill 执行）
- 量化分析：假设系统有 **30 个 Skills**，用户任务只需要 **2 个**，对比"全量加载"与"渐进式加载"的 Token 消耗差异
- 这种架构有什么潜在风险？如何缓解？

**2. Skill vs Tool 决策框架（15 分）**

在什么场景下应该使用 Skill 而非传统 Tool？请设计一套决策框架：
- 列出 3-5 个关键决策维度
- 针对以下场景给出选择建议并说明理由：
  - 文件读写操作
  - Web 搜索
  - 生成 Excel 报表
  - 生成 PowerPoint 演示文稿
  - 复杂的财务分析

**3. 跨模型兼容设计（20 分）**

你的系统需要支持多模型（Claude、GPT-4、Qwen 等）。Skills 是 Claude 特有功能。请设计一套架构实现：
- Claude 使用原生 Skills（最佳效果）
- 其他模型使用"降级方案"（Fallback）
- 保持统一的用户体验

需要说明：
- 核心组件的职责划分
- 降级方案的具体实现思路（如何让非 Claude 模型也能完成 Skill 任务）
- 如何监控和评估不同模型的 Skill 执行效果差异

## 题目四：创新能力综合考察


### 背景

现有的 LLM Agent 是"无状态"的——每次对话都从头开始，无法从历史交互中持续学习和优化。这是一个开放设计题，考察你的创新思维和系统设计能力。

### 核心挑战

设计一个 **"自适应学习型 Agent"**，能够从用户交互中持续学习并优化自己的行为：

1. **行为学习**：从用户反馈中学习（如"这个回答太长了"→ 未来回复更简洁）
2. **工具学习**：从工具使用经验中学习（如"这个 API 总是失败"→ 自动切换备选）
3. **知识积累**：从任务执行中积累领域知识（如"用户经常问财务报表"→ 预加载财务 Skill）

### 问题

**1. 系统架构设计（20 分）**

设计完整的系统架构，需要描述：
- **学习信号捕获**：从哪些来源获取学习信号？显式信号（用户评分）和隐式信号（行为分析）分别包括哪些？
- **知识表示与存储**：学习到的知识如何表示？如何存储？不同类型的知识（用户偏好/工具经验/领域知识）如何组织？
- **学习成果应用**：学习成果如何应用到实际对话中？对 Prompt、工具选择、响应策略分别有什么影响？

**2. 技术挑战分析（15 分）**

分析以下三个核心挑战，并提供解决思路（不需要写代码，描述核心策略即可）：

**挑战 A：灾难性遗忘**
- 学习新知识时可能忘记旧知识
- 如何在增量学习中保护核心知识？

**挑战 B：探索与利用平衡**
- 应该尝试新方法（探索）还是使用已验证方法（利用）？
- 如何设计动态平衡策略？

**挑战 C：错误学习防护**
- 用户反馈可能是错误的
- 如何验证学习信号的可靠性？如何处理矛盾的反馈？

**3. 创新点阐述（15 分）**

请提出 **至少一个** 你认为有价值的创新性设计，可以是：
- 新的学习机制或算法思路
- 新的知识表示方法
- 新的应用场景
- 与现有技术的创新结合（如与强化学习、知识图谱、联邦学习的结合）

需要说明：
- 创新点的核心思想
- 解决什么问题 / 带来什么价值
- 可能的实现路径和挑战
