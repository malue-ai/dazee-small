# AI Agent 自动化测试工程师面试题

> **适用岗位**：AI Agent 自动化测试工程师  
> **考察重点**：智能体测试策略、非确定性测试、质量评估、成本优化  
> **时间建议**：60 分钟  
> **特别说明**：聚焦 AI Agent 特有的测试挑战，而非传统软件测试

---

## 答题要求

1. **实现思路**：请用自然语言描述设计思路，鼓励给出测试框架代码结构或伪代码
2. **量化指标**：提出具体的量化指标和阈值
3. **工具选择**：可以推荐测试框架、工具库（如 pytest、locust、chaos engineering 工具等）
4. **实践经验**：欢迎结合 AI 系统或传统系统的测试经验

## 题目一：Agent 非确定性测试策略设计

### 背景

AI Agent 与传统软件系统的最大区别在于 **非确定性**：
- 相同输入，LLM 每次返回可能不同
- 传统的精确断言（`assert output == "expected"`）无法适用
- 但系统仍需保证质量和可靠性

### 场景描述

你负责测试一个企业级 AI Agent 系统，它具有以下核心功能：

**功能 1：任务规划（Planning）**
- 输入：用户需求（如"生成 Q4 销售分析报告"）
- 输出：分步计划（JSON 格式，包含 3-5 个步骤）
- 要求：计划应该合理、步骤完整、顺序正确

**功能 2：工具调用链（Tool Chain）**
- 输入：复杂任务（如"搜索 AI 行业报告并生成摘要"）
- 输出：工具调用序列（search → read → summarize）
- 要求：工具选择正确、调用顺序合理、参数有效

**功能 3：多轮对话（Conversation）**
- 输入：多轮用户消息
- 输出：连贯的回复，需要记住上下文
- 要求：不能遗忘之前的信息、不能自相矛盾

### 问题

**1. 测试策略设计（30 分）**

针对上述三个功能，设计完整的测试策略：

**任务规划测试**：
- 传统断言不可用（每次生成的 plan 文本不同），如何验证计划质量？
- 如何测试计划的"合理性"？提出至少 3 种验证方法
- 如何设计测试用例覆盖不同复杂度的任务？

**工具调用链测试**：
- 如何验证工具选择是否正确？
- 如何测试工具调用顺序的合理性？
- 如何处理"多条路径都正确"的情况？（如 search→summarize 和 search→read→summarize 都可以）

**多轮对话测试**：
- 如何测试 Agent 是否真正记住了上下文？
- 如何检测自相矛盾的回复？
- 如何设计对话测试用例？

**2. 质量评估方案（20 分）**

设计一套自动化的质量评估方案：

**评估维度**：
- 对于 Agent 的输出（文本、计划、工具调用），应该从哪些维度评估质量？
- 每个维度如何量化评分？

**LLM-as-Judge 方案**：
- 如何使用 LLM 来评估 Agent 输出的质量？
- 这种方案的优势和局限性是什么？
- 如何确保评估的稳定性和可靠性？

**成本控制**：
- LLM 调用成本高，如何在"测试覆盖率"和"成本"之间平衡？
- 哪些测试可以 Mock，哪些必须真实调用？

**3. 回归测试设计（15 分）**

Agent 系统迭代频繁（Prompt 调整、模型升级），如何设计回归测试？

- 如何建立"黄金测试集"？
- 如何检测性能退化？（不仅是 bug，还有质量下降）
- 如何处理"改进"与"退化"的边界判断？

## 题目二：端到端测试框架设计与多模型容灾测试

### 背景

生产级 AI Agent 系统需要完整的端到端测试，覆盖从用户请求到最终输出的全链路。同时，系统支持多模型容灾（Claude 主、Qwen 备），需要测试切换逻辑。

### 场景描述

系统架构：
```
用户请求 → API → ChatService → Agent → ModelRouter → LLM (Claude/Qwen)
                                    ↓
                                Tool Executor (10+ 工具)
                                    ↓
                                Memory Manager
```

关键特性：
1. **多模型容灾**：Claude 失败时自动切换 Qwen
2. **工具调用**：Agent 可调用 10+ 工具（搜索、文件、沙箱等）
3. **流式输出**：通过 SSE 推送增量响应
4. **状态管理**：Memory 保存对话历史和任务进度

### 问题

**1. E2E 测试框架设计（30 分）**

设计一个端到端测试框架，需要说明：

**测试环境隔离**：
- 如何隔离测试环境？（避免污染生产数据、Memory）
- 如何处理外部依赖？（LLM API、第三方搜索 API）
- 如何确保测试的可重复性？

**测试用例组织**：
- 如何设计测试用例结构？（单轮对话、多轮对话、复杂任务）
- 如何管理测试数据？（输入、预期输出、评估标准）
- 如何实现测试用例的复用和参数化？

**断言策略**：
- 对于流式输出（SSE），如何断言？
- 对于 Memory 状态，如何验证？
- 对于工具调用，如何验证调用顺序和参数？

**2. 多模型容灾测试（20 分）**

设计测试方案验证多模型容灾机制：

**故障注入测试**：
- 如何模拟 Claude API 失败？（超时、限流、服务不可用）
- 如何验证自动切换到 Qwen？
- 如何测试切换后的功能完整性？

**降级行为测试**：
- Claude Skills 在 Qwen 上降级处理，如何测试降级逻辑？
- 如何验证降级后用户体验的可接受性？
- 如何测试"切换回 Claude"的恢复逻辑？

**性能与成本测试**：
- 如何测试不同模型的响应速度差异？
- 如何测量切换过程的延迟？
- 如何验证成本优化效果？（Qwen 成本更低）

**3. 生产监控与测试联动（15 分）**

设计方案实现测试与生产监控的联动：

**生产探针测试**：
- 如何设计"健康探针"持续验证生产环境的 Agent 可用性？
- 探针测试与常规测试有什么区别？
- 如何在不影响真实用户的前提下进行探测？

**故障复现**：
- 生产环境发现问题（如某个任务失败），如何快速复现到测试环境？
- 如何记录和回放完整的请求上下文？

**质量看板**：
- 应该监控哪些核心指标？（成功率、响应时长、成本、质量分）
- 如何设计告警规则？
