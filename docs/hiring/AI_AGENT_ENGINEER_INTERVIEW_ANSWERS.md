## 题目一：Context Engineering（上下文工程）

### 1. 问题诊断（10 分）

**量化分析要点**：

| 维度 | 分析 |
|------|------|
| **Token 消耗** | 搜索 20KB + 分析 50KB + 浏览器 500KB ≈ 570KB ≈ 150,000+ tokens |
| **成本** | 单次对话可能消耗 $5-10（按 Claude Sonnet 定价） |
| **注意力问题** | 真正有用的信息被噪音淹没，LLM 可能"忘记"最初目标，任务完成率下降 |
| **Cache 失效** | 工具结果每次不同，无法利用 Prompt Caching |
| **延迟** | 传输和处理大 Context 增加 2-5 秒延迟 |
| **溢出风险** | 可能超过 Claude 200K / GPT-4 128K 限制 |

**加分点**：
- 能提到"注意力机制的长程依赖问题"
- 能使用实际 API 定价进行成本估算
- 能结合经验说明 Context 过大导致的任务完成率下降比例

---

### 2. 方案设计（25 分）

**Context Reduction（10 分）**

核心原则：**"返回指针而非内容"**

| 工具类型 | 精简策略 | 说明 |
|---------|---------|------|
| 搜索工具 | REFERENCE | 只返回 URL + 200 字符摘要，不返回全文 |
| 数据分析 | STRUCTURED | 只返回关键指标（均值、总数等），完整数据写入文件 |
| 浏览器 | REFERENCE | 只返回 URL + 页面标题，不返回 HTML |
| 文件读取 | TRUNCATE | 截断前 5000 字符，提示"使用 file_read 获取完整内容" |

关键设计：
- 每个精简结果必须包含 **access_hint**，指导 LLM 如何获取完整信息
- 精简前后需要记录日志，监控精简比例
- 对于 <10KB 的小结果，不精简直接返回

**Context Offloading（10 分）**

三层工具抽象架构：

| 层级 | 定位 | 适用场景 | 示例 |
|------|------|---------|------|
| **Level 1** | Function Calling | 高频、单一操作 | plan_todo, file_read, web_search |
| **Level 2** | Shell Utilities | 数据处理、灵活操作 | 通过 bash 调用 jq, grep, awk |
| **Level 3** | Python Scripts | 复杂编排、链式调用 | 在沙箱中执行多步骤数据处理 |

核心思想：
- Level 1 工具数量控制在 **5-8 个**（避免 LLM 选择困难）
- 大输出任务下沉到 Level 2/3，结果写入文件而非返回
- 复杂编排使用 Python 脚本，减少 LLM 调用轮次

**Cache Optimization（5 分）**

System Prompt 分块策略：

| 块 | 内容 | 是否缓存 |
|----|------|---------|
| Block 1 | 核心原则（不变） | ✅ 缓存 |
| Block 2 | 能力分类（很少变） | ✅ 缓存 |
| Block 3 | 核心工具（稳定） | ✅ 缓存 |
| Block 4 | 动态工具（按任务选择） | ❌ 不缓存 |

原则：**稳定内容在前，动态内容在后**；目标 Cache 命中率 >80%

---

### 3. 权衡分析（15 分）

**核心矛盾**：精简 Context → 可能增加 LLM 调用轮次

**平衡策略**：

1. **渐进式加载（Progressive Disclosure）**
   - 第一轮只返回摘要/引用
   - LLM 自主决定需要哪些完整内容
   - 按需加载，而非预加载

2. **场景化策略选择**
   - 生成型任务（写报告）：保留更多细节
   - 分析型任务（做决策）：只保留结构化数据
   - 问答型任务（回答问题）：激进精简 + 按需补充

3. **ROI 量化**
   - 典型场景：增加 2 轮调用（约 $0.02），节省 100K tokens（约 $0.30）
   - ROI > 10x，说明精简策略收益显著
   - 需要监控：任务完成率、用户满意度、平均轮次

**加分点**：
- 能提出具体的动态策略选择算法
- 能考虑用户体验（延迟感知）
- 能给出实际数据支撑

## 题目二：RAG/Memory 机制优化

### 1. 架构设计（20 分）

**三层记忆架构**：

| 层级 | 名称 | 内容 | 存储 | 生命周期 | 访问模式 |
|------|------|------|------|---------|---------|
| **Layer 1** | Working Memory | 当前对话消息、工具调用记录 | 内存 | 单次会话 | 直接读取 |
| **Layer 2** | Episodic Memory | 历史总结、用户偏好、显式记忆卡片 | 数据库 + 向量索引 | 用户级 | 语义检索 + 时间衰减 |
| **Layer 3** | Semantic Memory | 领域知识、技能文档、规则约束 | 向量数据库 | 系统级 | 纯语义检索 |

**记忆融合检索**：

1. Working Memory：直接返回最近 N 条消息
2. Episodic Memory：语义相似度 × 时间衰减权重
3. Semantic Memory：纯语义检索，按 domain 过滤
4. 融合排序：综合考虑相关性、时效性、置信度

**加分点**：
- 能提到具体的向量数据库选型（Qdrant、Milvus、Pinecone）
- 能描述时间衰减的具体公式或策略
- 能考虑记忆容量限制和清理策略

---

### 2. 核心问题解决（15 分）

**挑战 A："幻觉式遗忘"**

解决方案：**Memory-First Protocol**

核心原则：**NEVER trust thinking memory - ALWAYS read from Memory Tool**

强制协议：
- 每个步骤开始前 → **MANDATORY** 调用 get_plan/get_memory
- 禁止使用"我记得..."、"之前..."等推测性表述
- 所有状态来源必须是 Memory Tool 的返回

System Prompt 约束示例：
> ⚠️ CRITICAL: ASSUME INTERRUPTION  
> Your context window might reset at any moment.  
> 所有进度必须记录在 Memory 工具中，否则会丢失。  
> DO NOT 说"根据我的记忆..."  
> DO 每次操作前调用 memory.read()

**挑战 B：记忆冲突**

检测机制：
- 语义相似度检测：同一主题的新旧记忆
- 矛盾检测：使用 NLI（自然语言推理）模型判断
- 时间差分析：间隔越长，变化可能性越高

解决策略：

| 策略 | 适用场景 |
|------|---------|
| newest_wins | 偏好类信息（用户偏好可能变化） |
| explicit_first | 显式记忆优先于隐式记忆 |
| keep_both + 时间标记 | 事实类信息（可能并存） |
| ask_user | 重要冲突需要用户确认 |

关键点：**归档而非删除**——旧记忆移入归档区，保留回溯能力

### 3. Prompt 注入设计（15 分）

**记忆选择标准**：

1. **相关性**：与当前查询语义相关
2. **重要性**：高置信度、高频使用的记忆
3. **新鲜度**：近期更新的优先
4. **多样性**：覆盖不同维度（偏好、事实、约束）

选择算法思路：使用 **MMR（最大边际相关性）** 平衡相关性和多样性

**格式化结构**：

```
## User Context (Persona)

### Identity
- Role: 软件工程师（置信度: 0.9）

### Preferences  
- 喜欢简洁的报告格式
- 偏好 Python 编程语言

### Current Context
- 正在处理 Q4 销售数据分析

### Constraints
- 不要使用复杂的技术术语
```

**敏感信息处理**：

| 类型 | 处理方式 |
|------|---------|
| 密码、API Key | **拒绝保存**，直接返回错误 |
| 信用卡号、身份证 | **自动掩码**（如 **** **** **** 1234） |
| 邮箱、手机号 | **可选掩码**，根据配置决定 |

关键点：在 metadata 中标记过滤类型，保留审计日志

## 题目三：Agent Tools 实现机制

### 1. Progressive Disclosure 设计（15 分）

**实现流程**：

1. **Stage 1（发现阶段）**：加载所有 Skill 的简短描述（约 50 tokens/skill）
2. **LLM 决策**：基于用户查询和 Skill 描述，选择需要的 Skill
3. **Stage 2（加载阶段）**：仅加载选中 Skill 的完整内容（约 500-2000 tokens/skill）
4. **执行阶段**：按 SKILL.md 中的指令执行任务

**量化分析**：

| 方案 | 30 Skills 加载 | 2 Skills 完整 | 总计 | 节省 |
|------|---------------|---------------|------|------|
| **全量加载** | 30 × 1000 = 30,000 | - | 30,000 tokens | - |
| **渐进式加载** | 30 × 50 = 1,500 | 2 × 1000 = 2,000 | 3,500 tokens | **88%** |

加上 LLM 选择轮次（约 1,000 tokens），总计约 4,500 tokens，仍节省 **85%**

**潜在风险**：
- LLM 选择错误 Skill → 需要重新选择
- 增加一轮 LLM 调用的延迟
- 摘要不够准确导致漏选

**缓解措施**：
- 优化 Skill 摘要质量，包含清晰的能力标签
- 对高频 Skill 进行预加载
- 监控 Skill 选择准确率

---

### 2. Skill vs Tool 决策框架（15 分）

**关键决策维度**：

| 维度 | Tool | Skill |
|------|------|-------|
| 任务复杂度 | 单一操作 | 多步骤工作流 |
| 领域知识 | 通用操作 | 需要专业知识 |
| 输出类型 | 数据/文本 | 文件（Excel/PPT/PDF） |
| 配置复杂度 | 简单参数 | 复杂配置/模板/资源 |
| 使用频率 | 高频通用 | 低频专业 |

**场景选择**：

| 场景 | 选择 | 理由 |
|------|------|------|
| 文件读写 | **Tool** | 单一操作，无需领域知识 |
| Web 搜索 | **Tool** | 通用操作，高频使用 |
| 生成 Excel 报表 | **Skill** | 需要文件输出 + 图表/格式等专业能力 |
| 生成 PPT | **Skill** | 复杂配置（模板、布局）+ 文件输出 |
| 财务分析 | **Skill** | 多步骤 + 领域知识（财务比率、行业基准） |

---

### 3. 跨模型兼容设计（20 分）

**核心架构：统一工具调用器（UnifiedToolCaller）**

职责：
- 判断当前模型类型
- Claude → 使用原生 Skills API
- 其他模型 → 降级为 Fallback Tool

**降级方案实现思路**：

| Skill 类型 | 降级策略 |
|-----------|---------|
| **Pre-built Skills**（xlsx/pptx/pdf/docx） | 使用开源库替代（openpyxl、python-pptx） |
| **Custom Skills** | 将 SKILL.md 内容注入 System Prompt，作为指令执行 |

关键点：
- 结果格式必须统一，用户无感知
- 降级方案可能效果较差，需要标记和监控
- 如果任务强依赖 Skills，应优先使用 Claude

**监控指标**：

| 指标 | 说明 |
|------|------|
| Skill 使用率 | 各 Skill 的调用频次 |
| 降级率 | 使用 Fallback 的比例 |
| 成功率对比 | 原生 vs 降级的任务完成率 |
| 质量评分 | 用户对输出质量的反馈 |

**加分点**：
- 能描述具体的开源库替代方案
- 能设计自动切换逻辑（Claude 不可用时自动降级）
- 能考虑成本和效果的权衡

## 题目四：创新能力综合考察

### 1. 系统架构设计（20 分）

**学习信号捕获**：

| 类型 | 信号来源 | 示例 |
|------|---------|------|
| **显式信号** | 用户主动反馈 | 👍/👎 评分、"不对，应该是..."纠正、偏好设置 |
| **隐式信号** | 行为分析 | 重试次数、用户编辑输出、中断请求、工具成功/失败 |

**知识表示与存储**：

| 知识类型 | 表示方式 | 存储位置 |
|---------|---------|---------|
| 用户偏好 | 偏好向量 + 置信度 | 用户 Profile 表 |
| 工具经验 | 成功率、延迟、错误模式 | 工具统计表 |
| 领域知识 | 知识图谱 / 向量索引 | 知识库 |

**学习成果应用**：

| 应用场景 | 具体影响 |
|---------|---------|
| Prompt 调整 | 注入用户画像、调整响应长度/风格 |
| 工具选择 | 按可靠性排序、自动降级 |
| 策略优化 | 主动性调节、错误预防 |

### 2. 技术挑战分析（15 分）

**挑战 A：灾难性遗忘**

解决策略：
1. **弹性权重巩固（EWC）思想**：保护重要知识的权重
2. **记忆回放**：定期回顾历史学习样本
3. **分层存储**：核心知识（不可变）vs 临时知识（可覆盖）
4. **版本控制**：知识增量更新，保留历史版本

**挑战 B：探索与利用平衡**

解决策略：
1. **UCB（Upper Confidence Bound）算法**：
   - 未尝试过的方法有更高的"探索加分"
   - 随着尝试次数增加，探索加分降低
2. **ε-greedy 衰减**：
   - 初期高探索（ε=0.3）
   - 随经验积累，逐渐降低探索率（ε→0.01）
3. **上下文感知**：
   - 低风险任务允许更多探索
   - 高风险任务偏向利用已验证方法

**挑战 C：错误学习防护**

解决策略：
1. **一致性验证**：新学习与历史学习对比，不一致则隔离
2. **置信度阈值**：低置信度学习不更新核心知识
3. **人工审核**：重要变更队列，等待审核
4. **回滚机制**：发现错误学习可快速回滚

### 3. 创新点阐述（15 分）

**评分标准**：考察候选人的创新思维，以下为几个示例方向，考生可自由发挥

**示例方向 A：Meta-Learning 自适应 Prompt**

核心思想：不仅学习"用户偏好"，还学习"如何更好地提问"

观察：不同用户对不同 Prompt 风格响应不同
- 用户 A 喜欢结构化（1. 2. 3.）
- 用户 B 喜欢对话式
- 用户 C 需要更多示例

实现路径：使用 Thompson Sampling 或 Multi-armed Bandit 选择最佳 Prompt 模板

价值：提升用户满意度，减少无效沟通轮次

**示例方向 B：工具链自动优化**

核心思想：学习"工具组合模式"而非单个工具

观察：很多任务有固定的工具链
- 报告生成：search → analyze → write → format
- 数据处理：read → transform → validate → save

实现路径：记录成功的工具调用链，语义匹配推荐

价值：减少 LLM 推理负担，提高任务成功率

**示例方向 C：联邦学习隐私保护**

核心思想：在保护隐私的前提下，利用群体智慧

挑战：每个用户数据是私密的，但可能有共同模式

实现路径：
- 本地学习用户偏好
- 只分享模型更新（而非原始数据）
- 添加差分隐私噪声

价值：既保护隐私，又能利用群体数据提升效果

**加分点**：
- 创新点与实际问题紧密结合
- 能描述清晰的实现路径
- 能分析潜在挑战和局限性
- 能结合前沿论文或技术趋势
